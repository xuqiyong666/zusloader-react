# xxx-hub 主应用：混合应用目标与实现方案

## 1. 目标

- **独立开发**：每个子应用（React/Vue）独立工程，可单独启动、调试、发版。
- **统一体验**：交付后由主应用整合，用户在单一产品界面内使用，布局与导航由主应用统一。
- **可控集成**：子应用以明确产物形态接入；静态资源尽量与主应用同源或受控路径，降低跨域与部署复杂度。
- **可发现、可跳转**：子应用开发者知晓其它子应用与页面存在时，可通过主应用提供的 API 发起跨子应用跳转；目标不存在时中止并给出明确控制台提示。

## 2. 整体架构

- **1 个主应用 + N 个子应用**。
- 主应用在运行时暴露全局门面 **`window.__MainApp__`**（命名需全项目统一大小写，避免 `__MainAPP__` 类笔误）。
- 子应用在**生产**对接真实 `__MainApp__`；在**本地开发**提供**伪造/简化的 `__MainApp__`**，保证子应用可独立运行。

## 3. 主应用：`window.__MainApp__` 能力

1. **子应用注册与发现**  
   登记子应用 key、入口、路由元数据等；供菜单、权限与跨应用跳转解析使用。

2. **全局状态**  
   使用 **Zustand**（或等价方案）提供共享状态（如用户信息），子应用通过 `__MainApp__` 订阅/读取，避免各子应用重复造用户态。

3. **路由跳转**  
   统一对外暴露「跳转到某子应用某页面」等能力；子应用不直接操作主应用内部路由实现细节。

4. **共享运行时库**  
   主应用提供 **React、react-dom、axios、lodash、dayjs** 等公共依赖，减少重复加载。  
   **建议**：不要零散挂载到 `window`，而是集中到 **`window.__MainApp__.shareLibs`**（或等价结构化字段），与子应用构建的 `externals` 策略配套。

## 4. 子应用形态

- 子应用仓库内仍是**完整**的 React/Vue 应用：自带开发用 **Router + Layout**，便于日常开发。
- 交付给主应用的侧重：**可挂载的页面/业务能力**，页面组件**通常不含主站级 Layout**（由主应用提供外壳）。
- 子应用内需编写**适配层**：仅依赖 `__MainApp__` 契约（类型与版本需单独约定），便于 mock 与联调。

## 5. 部署与加载（manifest）

- 主应用支持**子应用代码包上传**，解压至**指定目录**，得到稳定可访问的**加载路径**，以规避跨域问题。
- 后端提供接口返回 **manifest JSON**，描述全部子应用元数据，例如：名称、`key`、入口脚本/资源地址等。
- 主应用根据 manifest **按需加载**子应用资源；可与「清单驱动加载」模式对齐（类似独立构建产物 + manifest 描述 CSS/JS 顺序）。

## 6. 跨子应用跳转约定

- 通过 `__MainApp__` 提供的服务注册表做**发现**；若目标子应用或页面不存在，**中止跳转**并在 **console** 输出明确信息（如「页面无法获取」）。
- 需在规格中补充：**路由参数、权限、未登录行为**等边界。

## 7. 待拍板事项（落地前规格）

| 项 | 说明 |
|----|------|
| `__MainApp__` 契约 | TypeScript 类型定义、版本号与兼容策略 |
| 子应用产物格式 | ESM / UMD / 微前端框架选型或自研 loader |
| `shareLibs` 与 externals | 子应用打包不重复打 react 等，运行时从 `shareLibs` 解析，版本对齐策略 |
| manifest 字段 | 除名称、key、入口外，是否含权限、默认路由、依赖主应用最低版本等 |
| 开发环境 mock | 伪造 `__MainApp__` 需实现哪些方法（跳转、状态、登录态） |

## 8. 与参考实现的关系（alpha-builder / App Editor）

- **可借鉴点**：独立 Vite 构建入口、构建后生成 **manifest**、宿主侧 **loader 按 manifest 注入样式与脚本**、壳在全局注册固定 API 后由宿主调用挂载。
- **本方案增量**：主应用需提供 **`__MainApp__`（注册发现、Zustand 状态、路由、shareLibs）**；子应用双模式（独立 dev + 主应用集成）；后端 manifest 服务与**同源静态资源**策略。
